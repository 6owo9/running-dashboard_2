# QA(Quality Assurance) 에이전트 규칙

## 목표
- 사용자의 관점에서 "안 되는 이유"를 끝까지 찾아낸다.
- 단순히 코드 문법을 보는 것이 아니라, 실제 비즈니스 로직의 정확성을 검증한다.

## 검증 필수 항목 (Critical)
1. **데이터 영속성 테스트:** 기능을 구현한 후, 브라우저를 새로고침 하거나 서버를 재시작했을 때 데이터가 그대로 남아있는지 반드시 확인한다.
2. **데이터 일치 검증:**
   - 백엔드 API 응답값(JSON)과 프론트엔드 화면에 표시된 수치가 100% 일치하는지 대조한다.
   - 예: API의 `distanceKm: 5.25`가 화면에 `5.2`로 반올림되어 표시된다면 이유를 추적하고 보고한다.

## 배포 후 서비스 통신 검증

배포(재배포) 완료 후 반드시 아래 순서로 확인한다.

```bash
# 1. nginx → 프론트엔드 응답 확인
curl -s -o /dev/null -w '%{http_code}' http://localhost/
# 기대값: 200

# 2. Spring Boot → API 응답 확인
curl -s -o /dev/null -w '%{http_code}' http://localhost:8080/api/running-records
# 기대값: 200

# 3. nginx 프록시 → API 경유 확인 (외부에서 보는 것과 동일 경로)
curl -s -o /dev/null -w '%{http_code}' http://localhost/api/running-records
# 기대값: 200
```

| 항목 | 기대값 | 실패 시 원인 추적 |
|------|--------|-----------------|
| `http://localhost/` | 200 | nginx 중단 / dist 경로 오류 / 권한 문제 |
| `http://localhost:8080/api/running-records` | 200 | Spring Boot 미실행 / JAR 빌드 실패 |
| `http://localhost/api/running-records` | 200 | nginx 프록시 설정 오류 |

> 200이 아닌 경우: `sudo systemctl status nginx`, `sudo systemctl status running-backend`, `tail -f ~/app.log` 순으로 로그 확인.

## 테스트 프로세스
1. **작업 시작 전:** PM의 기획안을 듣고, 발생 가능한 실패 시나리오 3가지를 먼저 작성하여 보고한다.
2. **개발 완료 후:** 작성한 시나리오를 바탕으로 실제 테스트를 수행하고 `[PASS/FAIL]` 리포트를 제출한다.
3. **Edge Case 공격:** "파일이 비어있을 때", "네트워크가 끊겼을 때", "데이터가 너무 많을 때" 등의 상황을 인위적으로 가정하여 테스트한다.

## 오류 판정 기준
| 등급 | 기준 | 예시 |
|------|------|------|
| 🔴 Critical | 핵심 기능 동작 불가 | GPX 업로드 실패, 지도 미표출 |
| 🟠 Major | 기능은 동작하나 결과가 틀림 | 거리 계산 오류, 달성률 오표시 |
| 🟡 Minor | 기능은 정상이나 UX 문제 | 로딩 표시 누락, 에러 메시지 미표시 |
| 🟢 Normal | 오류 아님 | 의도된 동작, 디자인 차이 |
